## 1. 背景と目的

### 1.1 背景

蒼理塾(琉球サーティー)は、沖縄を拠点とする無償の学習支援コミュニティであり、活動はオフラインとオンラインの二層構造を持つ。

- **オフライン**:生徒と講師がたまに集まり、関係性・コミュニティとしての繋がりを維持する場
- **オンライン**:日々の学習の実体が動く場。宿題の付与と進捗確認、テスト・模試の実施と採点、成績の記録、次に学ぶべき内容の判断、スケジュール管理

現状、オンライン活動は Discord 上で運用されている。Discord はコミュニケーション基盤としては十分に機能しているが、以下の点で構造的な限界がある。

1. **状態が流れる**:課題の進捗・成績・締切といった「状態を持つ情報」が、チャットログという「流れる情報」の中に埋もれる
2. **横断的な把握が困難**:講師が「生徒Aの今週の課題消化率」「生徒Bの直近3回の模試推移」を一覧で見る手段がない
3. **蓄積が資産にならない**:成績データが構造化されていないため、「この点数推移ならば次はこれを学ぶべき」という類推・提案に繋げられない

### 1.2 目的

**学習者が効率よく、最適に学習できる状態を作ること。** そのために、Discord が苦手とする「状態の管理と蓄積」に特化したWebアプリを構築する。

本アプリは Discord を置き換えるものではない。**会話は Discord、状態はアプリ** という明確な役割分担を取る。

### 1.3 成功の定義

- 講師が1分以内に「全生徒の今週の進捗」を把握できる
- 生徒が迷わず「今日やるべきこと」に辿り着ける
- 成績データが構造化されて蓄積され、将来の学習提案機能(Phase 2)の土台になる

## 2. 設計原則(プロジェクトの憲法)

要件の追加・削除で迷ったとき、常にここに立ち返る。

| # | 原則 | 含意 |
|---|---|---|
| P-1 | **Discordで代替できる機能は作らない** | チャット、通話、雑談、お知らせ配信はDiscordの領分。アプリには持ち込まない |
| P-2 | **状態を持つ情報だけを扱う** | 課題の進捗、成績、予定。流れる情報(会話)は扱わない |
| P-3 | **MVPは小さく、データ設計は将来を見据えて** | 機能は最小限に絞るが、成績・進捗のデータ構造はPhase 2(学習提案)を見越して設計する |
| P-4 | **生徒の入力負荷を最小化する** | 生徒が「記録のための作業」を強いられた瞬間、使われなくなる。入力は1操作で完結させる |
| P-5 | **無償コミュニティとして持続可能なコスト** | 運用費は月額ほぼゼロ〜数百円に収める。有料SaaSへの依存を避ける |
| P-6 | **未成年の個人情報を預かる自覚を持つ** | 成績は機微情報。閲覧権限・データ保護は最初から設計に組み込む |

## 3. ステークホルダーとユーザー像

### 3.1 ロール定義

| ロール | 説明 | 主なユースケース |
|---|---|---|
| **生徒** | 学習支援を受ける中高生 | 今週の課題確認、進捗更新、自分の成績推移閲覧、予定確認 |
| **講師** | 課題を出し進捗を見守る大学生等 | 課題の作成・割当、担当生徒の進捗確認、成績の登録・閲覧 |
| **運営** | 蒼理塾の運営メンバー | ユーザー管理、全体の予定登録、全データの閲覧 |

### 3.2 代表的ペルソナ

**生徒(高2・Aさん)**:スマホしか持っていない。Discordは毎日見るが、固定ピン留めされた課題一覧は見落とす。「今日なにやればいいか」が一目で分かれば動ける。

**講師(大学生・Bさん)**:担当生徒は3〜5名。週次で課題を出すが、誰がどこまで終わったかをDiscordのDMで個別確認しており、確認自体が負担になっている。

**運営(Cさん)**:模試日やオフライン会の日程を全員に周知したい。また、コミュニティ全体としての活動実態(課題消化率、成績の動き)を把握し、講師配置や支援方針の判断材料にしたい。

## 4. スコープ定義

### 4.1 フェーズ構成

| フェーズ | テーマ | 内容 |
|---|---|---|
| **Phase 1(MVP)** | 状態の可視化 | 課題タスク管理、模試・テスト成績登録、カレンダー、ユーザー管理 |
| **Phase 2** | 蓄積からの示唆 | 成績推移の分析、弱点分野の可視化、次に学ぶべき内容の提案(ルールベース→AI) |
| **Phase 3** | 学習の社会化 | Studyplus的な学習時間の記録・相互閲覧、励まし機能(※需要を再検証してから着手) |

### 4.2 MVPでやらないこと(明示的なスコープ外)

スコープ外を明文化することが、たたき台の最重要機能である。

- ❌ チャット・メッセージ機能(→ Discord)
- ❌ お知らせ・通知の配信基盤(→ Discord。ただしDiscord Webhookによる自動通知連携は許容、後述)
- ❌ 学習時間の記録・タイマー(→ Phase 3で需要を再検証)
- ❌ 教材の配布・ファイル管理(→ Discord / Google Drive)
- ❌ オンライン上でのテスト実施機能(採点は人間が行い、**結果だけ**をアプリに登録する)
- ❌ 保護者向け画面(将来検討)
- ❌ ネイティブアプリ(スマホ対応はレスポンシブWebで実現)

## 5. 機能要件(Phase 1 / MVP)

優先度は MoSCoW 法(Must / Should / Could)で表記。

### F-1 課題・タスク管理

週次の課題をタスク形式で管理し、進捗を生徒・講師の双方から可視化する。MVPの中核。

| ID | 要件 | 優先度 |
|---|---|---|
| F-1-1 | 講師は課題を作成できる(タイトル、説明、科目、締切日、割当先生徒) | Must |
| F-1-2 | 課題は複数の生徒に一括で割当できる(同じ課題を担当生徒全員へ) | Must |
| F-1-3 | 生徒は自分の課題一覧を「今週」「期限切れ」「完了済み」で閲覧できる | Must |
| F-1-4 | 生徒は課題のステータスを更新できる:`未着手 → 進行中 → 完了` の3状態 | Must |
| F-1-5 | ステータス更新は1タップで完了する(P-4:入力負荷の最小化) | Must |
| F-1-6 | 講師は担当生徒全員の課題進捗を一覧(ダッシュボード)で確認できる | Must |
| F-1-7 | 課題に進捗率(例:問題集 p.10〜30 のうち p.22 まで)をメモできる | Should |
| F-1-8 | 締切超過の課題は視覚的に区別される(色・並び順) | Should |
| F-1-9 | 課題テンプレート(毎週同じ形式の課題を複製) | Could |
| F-1-10 | 生徒から講師への一言コメント(「ここが分からなかった」等。※議論はDiscordへ誘導) | Could |

**ステータスを3状態に絞る理由**:5状態以上にすると生徒が更新しなくなる。「進行中」の存在は、講師が「手は付けている」ことを把握できる最小限の中間状態として必要。

### F-2 模試・テスト成績管理

成績を構造化して登録・蓄積し、推移を可視化する。**Phase 2 の学習提案は、ここのデータ設計の質に全面的に依存する。**

| ID | 要件 | 優先度 |
|---|---|---|
| F-2-1 | 講師または生徒は、テスト結果を登録できる(テスト名、実施日、種別) | Must |
| F-2-2 | 成績は**科目単位**で登録する(国数英理社+細分科目。点数、満点、可能なら偏差値) | Must |
| F-2-3 | テスト種別を区別する:`模試` / `定期テスト` / `小テスト(確認テスト)` | Must |
| F-2-4 | 生徒は自分の成績推移を時系列グラフで閲覧できる(科目別) | Must |
| F-2-5 | 講師は担当生徒の成績推移を閲覧できる | Must |
| F-2-6 | 生徒は**他の生徒の成績を閲覧できない**(権限の厳格な分離) | Must |
| F-2-7 | 模試の場合、志望校判定(A〜E)を任意項目として記録できる | Should |
| F-2-8 | 科目内の**分野タグ**(例:数学>二次関数、英語>長文読解)を任意で付与できる | Should |
| F-2-9 | 成績登録時に振り返りメモ(自由記述)を残せる | Could |

**F-2-8 が将来の鍵**:分野タグの蓄積があって初めて「二次関数で連続して失点 → 次週の課題は二次関数の基礎に戻る」というPhase 2の提案ロジックが成立する。MVPでは任意項目に留めるが、データモデルには最初から組み込む(P-3)。

### F-3 カレンダー

締切・模試日・オフライン会など、コミュニティの時間軸を一望できるシンプルなカレンダー。

| ID | 要件 | 優先度 |
|---|---|---|
| F-3-1 | 月表示カレンダーで予定を閲覧できる | Must |
| F-3-2 | 課題の締切日は自動的にカレンダーへ反映される(二重入力させない) | Must |
| F-3-3 | 運営・講師は予定を登録できる(模試日、オフライン会、面談など) | Must |
| F-3-4 | 予定の公開範囲を選べる:全体 / 特定生徒のみ | Should |
| F-3-5 | 週表示・リスト表示の切替 | Could |

### F-4 ユーザー管理・認証

| ID | 要件 | 優先度 |
|---|---|---|
| F-4-1 | メールアドレスによるログイン認証 | Must |
| F-4-2 | ロール(生徒/講師/運営)に基づくアクセス制御 | Must |
| F-4-3 | 講師と生徒の担当関係(1講師:N生徒)を設定できる | Must |
| F-4-4 | 新規登録は**招待制**(運営が発行する招待リンク経由のみ。野良登録は不可) | Must |
| F-4-5 | 運営はユーザーの無効化(退会処理)ができる | Must |
| F-4-6 | プロフィールは最小限(表示名、学年、所属)。本名・住所等は収集しない | Must |

### F-5 Discord連携(任意・軽量)

P-1 に反しない範囲で、Discordを「通知の出口」として活用する。

| ID | 要件 | 優先度 |
|---|---|---|
| F-5-1 | 課題の新規割当時、Discord Webhookで対象チャンネルに通知を飛ばせる | Should |
| F-5-2 | 締切前日のリマインド通知 | Could |

## 6. 非機能要件

| 分類 | ID | 要件 |
|---|---|---|
| 性能 | NF-1 | 主要画面の初期表示は3秒以内(沖縄のモバイル回線を想定) |
| 対応端末 | NF-2 | スマホファースト(生徒の主端末)。PC・タブレットにもレスポンシブ対応 |
| 可用性 | NF-3 | 個人開発規模として常識的な範囲(99%目安)。計画停止は事前にDiscordで周知 |
| セキュリティ | NF-4 | 全通信のHTTPS化。パスワードはハッシュ化保存(認証基盤に委譲) |
| セキュリティ | NF-5 | データベースレベルでの行単位アクセス制御(Row Level Security)。アプリのバグでも他生徒の成績が漏れない構造にする |
| プライバシー | NF-6 | 利用開始時に、収集データと利用目的を平易な日本語で説明し同意を得る。未成年については保護者の認知を前提とした運用ルールを定める |
| プライバシー | NF-7 | 退会時、本人データは合理的な期間内に削除可能とする |
| コスト | NF-8 | 月額運用費の上限目安:0〜1,000円(無料枠中心の構成) |
| 保守性 | NF-9 | 運営メンバーが引き継げるよう、最低限のREADME(環境構築・デプロイ手順)を整備する |

## 7. データモデル(概念設計)

Phase 2 を見据えた骨格。MVP実装時はこのうちMust機能に必要なテーブルから着手する。

```
users(ユーザー)
  id / 表示名 / ロール(生徒・講師・運営) / 学年 / 状態(有効・無効)

assignments(課題マスタ)
  id / 作成者(講師) / タイトル / 説明 / 科目 / 締切日

assignment_tasks(課題×生徒 = タスク)
  id / 課題id / 生徒id / ステータス(未着手・進行中・完了) / 進捗メモ / 完了日時

exams(テストマスタ)
  id / テスト名 / 実施日 / 種別(模試・定期・小テスト)

exam_scores(成績)
  id / テストid / 生徒id / 科目 / 得点 / 満点 / 偏差値(任意) / 判定(任意) / メモ

score_topics(分野タグ ※Phase 2の布石)
  id / 成績id / 分野名 / 得点 / 満点

events(予定)
  id / タイトル / 日時 / 種別(模試・オフライン会・その他) / 公開範囲

mentorships(担当関係)
  講師id / 生徒id
```

設計上のポイント:課題(assignments)とタスク(assignment_tasks)を分離することで「同じ課題を複数生徒に割当」(F-1-2)が自然に表現でき、生徒ごとの進捗が独立に管理できる。

## 8. 画面一覧(MVP)

| # | 画面 | 主ユーザー | 概要 |
|---|---|---|---|
| 1 | ログイン / 招待登録 | 全員 | 招待リンクからの初回登録、ログイン |
| 2 | 生徒ホーム | 生徒 | 「今日やること」起点。今週の課題と直近の予定が一画面に収まる |
| 3 | 課題詳細 | 生徒 | ステータス更新(1タップ)、進捗メモ |
| 4 | マイ成績 | 生徒 | 科目別の成績推移グラフ、過去の成績一覧 |
| 5 | 講師ダッシュボード | 講師 | 担当生徒×課題のマトリクス。進捗が色で一望できる |
| 6 | 課題作成・割当 | 講師 | 課題の作成と生徒への一括割当 |
| 7 | 成績登録 | 講師・生徒 | テスト結果の入力(科目ごとの一括入力フォーム) |
| 8 | 生徒詳細(講師視点) | 講師 | 特定生徒の課題履歴+成績推移を縦断的に閲覧 |
| 9 | カレンダー | 全員 | 月表示。締切・予定の一覧 |
| 10 | 管理画面 | 運営 | ユーザー管理、招待発行、担当関係の設定、予定登録 |

画面数は10。これ以上増える提案が出たら、P-1〜P-3に照らして一度疑うこと。

## 9. 技術スタック(推奨案)

| レイヤ | 技術 | 選定理由 |
|---|---|---|
| フロント/フルスタック | Next.js | 開発メンバーの既存知見を活かせる。レスポンシブWebでスマホ対応 |
| BaaS | Supabase | 認証・DB・Row Level Security(NF-5)を無料枠で賄える。招待制認証も実装容易 |
| ホスティング | Vercel | 無料枠。デプロイ自動化 |
| グラフ描画 | Recharts 等 | 成績推移グラフ(F-2-4) |
| 通知連携 | Discord Webhook | F-5。サーバー追加コストゼロ |

月額コスト試算:ユーザー数十名規模なら **0円**(全サービス無料枠内)。NF-8 を満たす。

## 10. 開発ロードマップ(目安)

| 期間 | マイルストーン |
|---|---|
| Week 1–2 | データモデル確定、認証・ユーザー管理(F-4) |
| Week 3–4 | 課題・タスク管理(F-1 のMust)、講師ダッシュボード |
| Week 5 | 成績登録・推移グラフ(F-2 のMust) |
| Week 6 | カレンダー(F-3)、Discord通知(F-5-1) |
| Week 7 | 運営・講師数名+生徒数名でクローズドβ |
| Week 8 | フィードバック反映 → 蒼理塾全体へ展開 |

βテストの観点:**「生徒が言われなくてもステータスを更新するか」**が最大の検証項目。更新されないなら、機能を足すのではなく入力導線を削る方向で直す(P-4)。

## 11. KPI(MVPの成功指標)

| 指標 | 目標 |
|---|---|
| 週次アクティブ生徒率 | 70%以上(週1回以上アプリを開く) |
| 課題ステータス更新率 | 割当課題の80%以上が締切までに何らかの状態更新を受ける |
| 成績登録の継続率 | 模試・定期テスト実施後2週間以内の登録率 90% |
| 講師の確認時間 | 進捗確認にかかる時間が「DMで個別確認」比で体感半減 |

## 12. リスクと対策

| リスク | 影響 | 対策 |
|---|---|---|
| 生徒が更新しなくなる(最大リスク) | アプリが形骸化 | 1タップ更新の徹底(F-1-5)、Discord通知での想起(F-5)、オフライン会での利用習慣づけ |
| Discordとの役割が曖昧化し機能が肥大 | 開発・運用負荷増 | 設計原則P-1/P-2を判断基準として明文化済み。新機能提案は必ず原則と照合 |
| 成績データの漏洩 | コミュニティの信頼喪失 | RLSによるDB層での権限分離(NF-5)、収集データの最小化(F-4-6) |
| 開発者の多忙による停滞 | プロジェクト休眠 | MVPスコープの厳守、引き継ぎ可能なドキュメント整備(NF-9) |
| Phase 3(学習時間共有)への期待先行 | スコープ膨張 | 「Discordで既に十分なら不要」の初期判断を尊重し、MVP定着後に需要を再調査 |

## 13. 未決事項(運営での要議論)

1. 成績の登録者は誰か:講師が登録するか、生徒の自己申告か、併用か(本書は併用を仮置き)
2. 偏差値・判定の扱い:生徒のモチベーションへの影響を踏まえ、本人にどこまで見せるか
3. 招待制の運用フロー:新規生徒の参加時、誰がいつ招待リンクを発行するか
4. 保護者への説明:同意取得の具体的な方法(書面/口頭/Discord周知)
5. Phase 2 の着手判断基準:成績データが何件たまったら学習提案に着手するか
