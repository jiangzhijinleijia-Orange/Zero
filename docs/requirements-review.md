## 1. 総評

たたき台として完成度は高い。特に以下は強みであり、レビューで崩さないこと。

- **スコープ外の明文化(§4.2)と設計原則(§2)**:機能肥大への防波堤が最初からある

一方で、**「決めずに進むと後で高くつく」論点**がいくつか残っている。本書では §13 の5項目に推奨案を付け(→ §2)、それ以外に v1.1 までに決めるべき論点を優先度付きで挙げる(→ §3)。

## 2. 未決事項(§13)への推奨案

### 2-1. 成績の登録者は誰か

**推奨:併用を維持しつつ、テスト種別ごとに「既定の登録者」を定める。**

| 種別 | 既定の登録者 | 理由 |
|---|---|---|
| 模試・定期テスト | 生徒(自己申告) | 結果を最初に手にするのは本人。講師経由は転記の遅延・負担を生む |
| 小テスト(確認テスト) | 講師 | 採点者が講師であり、採点と同時に登録するのが最短(P-4) |

講師はどの種別でも代理登録・修正できることとする。あわせて `exam_scores` に **登録者(registered_by)** を記録する列を追加する。誰が入れたかが残らないと、KPI「成績登録の継続率 90%」の未達時に原因(生徒が入れないのか、講師が入れないのか)を切り分けられない。

### 2-2. 偏差値・判定の扱い

**推奨:MVPでは「本人のデータは本人にすべて見せる」を原則とする。**

理由:

1. 模試・定期を生徒の自己申告にするなら(2-1)、本人は登録時点で値を知っている。アプリで隠す意味がない
2. 「本人にも一部隠す」設計は表示権限の分岐を増やし、MVPの複雑度を上げる(P-3 に反する)
3. モチベーションへの配慮は**隠すのではなく見せ方で行う**:推移グラフは点数・偏差値の「伸び」を主役にし、志望校判定(A〜E)はグラフ化せず一覧表示に留める

なお「他の生徒の成績は見えない」(F-2-6)は当然維持。順位表示・生徒間比較は Phase 3 以降も含めて行わない方針を v1.1 に明記することを推奨。

### 2-3. 招待制の運用フロー

**推奨:発行は運営のみ。「ロールと担当講師を指定した招待リンク」を発行する。**

フロー案:

1. Discord 上で新規生徒の参加が確定する
2. 運営が管理画面で招待を発行。このとき **ロール(生徒/講師)と、生徒の場合は担当講師を指定**する
3. 招待リンクを Discord の DM で本人に送付
4. 本人がリンクから登録(表示名・学年のみ入力)。**有効期限 72 時間・1回限り有効**

ポイントは手順2:招待時に担当関係(F-4-3)まで決めてしまうことで、登録直後から課題割当が可能になり、「登録したが何も表示されない」という初回離脱を防げる。発行権限を講師へ広げるかは、運営の手が回らなくなった時点で再検討すれば足りる。

### 2-4. 保護者への説明・同意取得

**推奨:Discord 周知のみは避け、記録が残る形式(電子フォームまたは書面)で同意を取得する。**

NF-6 は「保護者の認知を前提とした運用」を求めており、未成年の成績という機微情報を扱う以上、**同意の証跡**が必要。口頭・Discord 周知では後から確認できない。

- 収集データ・利用目的・閲覧できる人・削除方法を平易な日本語で記した **1枚ものの説明文書**を用意する
- 同意は Google Forms 等の電子フォーム(無料、記録が残る、NF-8 に適合)を基本とし、希望者には書面でも受け付ける
- 同意記録の保管責任者(運営の誰か)を決めておく

これはアプリの機能要件ではなく**運用要件**なので、開発と並行して準備できる。クローズドβ(Week 7)開始までに整っていればよい。

### 2-5. Phase 2 の着手判断基準

**推奨:データの「件数」ではなく「質と定着」で条件を置く。**

Phase 2 の提案ロジックは分野タグ(F-2-8)に全面依存すると要件定義書自身が述べている。タグなしの成績が何千件あっても提案はできない。判断基準の案:

1. アクティブ生徒の 70% 以上が、**3回分以上**のテスト成績を持つ(推移が語れる最低ライン)
2. 直近の成績登録のうち **50% 以上に分野タグが付いている**(タグ運用が定着している)
3. MVP の KPI(§11)を 4 週連続で達成している(土台のアプリが使われている)

3条件が揃った時点で、まず**ルールベース**(例:同一分野で2回連続して得点率が閾値未満 → その分野の基礎課題を提案)から着手する。AI 活用はルールベースの提案が現場で「当たる」と確認できてからでよい。

## 3. 追加の論点(v1.0 に明記がないもの)

優先度:**[A] v1.1 までに要決定**(設計・実装の手戻りに直結)/ **[B] β開始までに決定**/ **[C] 記載の明確化のみ**

### [A-1] 認証方式:メールアドレスは生徒に届くか

F-4-1 はメールログインを Must とするが、ペルソナ(§3.2)の生徒像は「スマホしか持たない、Discord は毎日見る」。**中高生がメールを日常的に確認するかは疑わしく**、パスワードリセットや招待リンクの送付(メール経由の場合)が最初の脱落ポイントになり得る。これは §12 の最大リスク「生徒が更新しなくなる」に直結する。

**推奨:Discord OAuth ログインを第一候補として検討する。** Supabase は Discord 認証を標準サポートしており実装コストは低い。P-1(Discord で代替できる機能は作らない)は「チャット等の機能」の話であり、認証に Discord アカウントを使うことはむしろ生徒の導線として自然。メールログインは講師・運営向けの代替手段として残す構成もあり得る。

### [A-2] 科目・分野タグのマスタ化(表記ゆれ対策)

F-2-2「国数英理社+細分科目」の具体リストが未定義。科目を自由入力にすると「数学/数I/数1」のような表記ゆれが必ず発生し、**科目別推移グラフ(F-2-4)と Phase 2 の分析が成立しなくなる**。

**推奨:**

- 科目は**マスタ管理(選択式)**とし、初期リスト(例:英語・数学I/A・数学II/B・現代文・古典・物理・化学・生物・地理・歴史・公民…)を v1.1 の付録として確定する
- 分野タグ(F-2-8)も自由入力にせず、**運営・講師のみが語彙を追加できる選択式**とする。タグは Phase 2 の生命線であり、ここの統制を怠ると蓄積が資産にならない(§1.1 の問題意識の再来)

### [A-3] 講師の閲覧範囲と、担当変更時の扱い

F-2-5 は「担当生徒の成績」とあるが、**講師は担当外の生徒のデータを一切見られない、でよいか**を明文化したい。RLS(NF-5)のポリシー設計に直結し、後から変えるとセキュリティ設計のやり直しになる。

あわせて**担当変更**(講師の交代・卒業)時の扱いも決めたい。推奨:`mentorships` に期間(開始日・終了日)を持たせ、現担当は生徒の全履歴(過去の担当期間分も含む)を閲覧できる、とする。引き継いだ講師が過去の推移を見られないと指導が成り立たないため。

### [A-4] 担当関係の多重度:生徒は複数の講師を持てるか

F-4-3 は「1講師:N生徒」とあるが、**1人の生徒に科目別で複数の講師が付くケース**(数学は B さん、英語は C さん)は現実に起こりそう。データモデルの `mentorships` は多対多を許す形になっており、要件文とモデルが食い違っている。

**推奨:多対多(生徒は複数講師を持てる)を正とし、v1.1 の F-4-3 の文言を直す。** 実装コストはほぼ変わらない。科目単位の担当までモデル化するかは、MVP では持たない(担当講師は生徒の全科目を見られる)で十分。

### [B-1] 模試の「提供元」と基準差

模試は外部事業者(進研・河合・駿台等)のもので、満点・偏差値の母集団が異なる。`exams` に**提供元**を持たないと、異なる模試の偏差値を同一グラフに並べて誤読する恐れがある。推奨:`exams` に提供元(任意項目)を追加し、推移グラフでは種別・提供元をまたぐ比較に注記を出す程度の軽い対応に留める。

### [B-2] 退会時削除(NF-7)の具体化

「本人データは削除可能」とあるが、削除の範囲を決めたい。推奨:成績・課題履歴は**物理削除**(無償コミュニティとして匿名化保持の必然性が薄く、説明もしやすい)。削除までの「合理的な期間」は 30 日を目安に明記する。

### [B-3] 学年の進級・卒業の扱い

`users` に学年があるが、年度替わりの進級処理(高2→高3)と卒業生の扱い(無効化?閲覧のみ?)が未定義。MVP では「運営が管理画面で手動更新」で十分だが、その一文を v1.1 に入れておくと迷わない。

### [B-4] KPI の計測手段

§11 の KPI(週次アクティブ率など)を測る仕組みが機能要件に含まれていない。専用の分析基盤は不要(P-5)だが、**最終ログイン日時とステータス更新日時が記録されていれば全 KPI は SQL で集計できる**。データモデルに最低限のタイムスタンプを持たせることを実装方針として明記する。

### [C-1] カレンダーの表示範囲の確認

F-3-2 で課題締切が自動反映されるが、生徒のカレンダーに出るのは**自分に割り当てられた課題の締切のみ**(他生徒の締切は見えない)であることを明文化したい。F-2-6 の権限分離と整合する話で、実装者への指示として書いておくだけでよい。

### [C-2] F-1-7 の表現

「進捗率」とあるが要件の実体は自由記述メモ(p.22 まで等)。数値の進捗率(%)と誤読されないよう「進捗メモ」に統一することを推奨。データモデル(`assignment_tasks` の進捗メモ)とも揃う。

## 4. データモデルへの反映まとめ(v1.1 で更新)

§2・§3 の推奨を採る場合、概念モデル(§7)への影響は以下のとおり。いずれも小さい。

| テーブル | 変更 | 根拠 |
|---|---|---|
| exam_scores | 登録者(registered_by)を追加 | 2-1 |
| subjects(新設) | 科目マスタ | A-2 |
| topics(新設) | 分野タグの語彙マスタ(score_topics はこれを参照) | A-2 |
| mentorships | 開始日・終了日を追加。多重度は N:N と明記 | A-3, A-4 |
| exams | 提供元(任意)を追加 | B-1 |
| users | 最終ログイン日時(KPI 集計用) | B-4 |

## 5. レビュー会議のアジェンダ案(60分)

1. 設計原則(§2)とスコープ外(§4.2)の承認 — 10分(ここが固まれば以降が速い)
2. 未決事項への推奨案(本書 §2)の採否 — 20分
3. 追加論点 [A-1]〜[A-4] の決定 — 20分
4. [B] 項目の担当者と期限の割り振り — 5分
5. v1.1 更新の担当と期日 — 5分

[C] 項目は会議で扱わず、v1.1 更新時に文言修正で吸収してよい。
